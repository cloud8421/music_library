## Description

<!-- SECTION:DESCRIPTION:BEGIN -->

The root routing model already supports progressive disclosure, but the mandatory architecture reference is large and several skill descriptions activate on broad, overlapping vocabulary. Preserve AGENTS.md as the stable entry point while splitting detailed architecture references and tightening skill triggers so each journey loads only the context needed for that domain.
<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria

<!-- AC:BEGIN -->

- [ ] #2 AGENTS.md retains its required architecture-first rule but names which focused reference to load for web, data, background, integration, production, and Pi harness work
- [ ] #3 Skill descriptions use narrow action and subsystem triggers that avoid loading unrelated large skills for generic words such as error, worker, query, or performance

## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->

## Objective and chosen route

Preserve `AGENTS.md` as the stable routing layer and `docs/architecture.md` as the required first read, but make that first read a concise map. Move detailed, independently changing tables into focused architecture references and shorten skill descriptions to precise action/subsystem triggers. Do not delete architecture facts or duplicate them across the new files.

The rejected alternatives are making architecture optional, which would weaken boundary awareness, and merely shortening prose in place, which would soon regrow because unrelated domains still share one file.

## Implementation steps

1. Capture a reproducible baseline: byte and approximate token counts for `AGENTS.md`, `docs/architecture.md`, all skill descriptions, and the full SKILL files loaded by representative coding, UI, database, worker, production, API, testing, and documentation journeys.
2. Define non-overlapping architecture ownership before moving text. Keep the overview, supervision boundaries, repo/context map, cross-cutting conventions, and links in `docs/architecture.md`. Move detailed data/search material to `docs/architecture/data.md`, workers/schedules to `docs/architecture/background.md`, routes/LiveViews/frontend to `docs/architecture/web.md`, external APIs to `docs/architecture/integrations.md`, and development/Pi tooling to `docs/architecture/tooling.md` or `docs/pi-harness.md` where operational rather than architectural.
3. Move existing sections without rewriting facts first, then remove duplicates. Add a contents/routing table to the root map that tells an agent exactly which focused reference is required for each kind of change. Preserve the maintenance rule in every owned reference.
4. Update `AGENTS.md` so the mandatory first step still reads the concise root map, then directs the agent to focused references based on touched subsystem. Keep production, task, conventions, usage-rules, and Presto routing intact.
5. Audit every skill frontmatter description. Replace broad trigger words with concrete user intents and code surfaces; for example, generic “error” must not alone activate production investigation, generic “worker” must not mean the Pi worker epoch, and generic “performance” must not load both database and production skills without a matching subsystem. Keep descriptions within the Agent Skills limit.
6. Create a context-routing fixture/table of representative prompts and expected required references/skills. Add lightweight structural checks for broken links, duplicate architecture headings/owners, frontmatter validity, and description size. Do not attempt nondeterministic model-based trigger tests in CI.
7. Re-run the inventory. Require the root architecture first-read payload to fall by at least 50% and the always-present skill-description payload to fall materially without dropping any available skill. For each journey, verify only the root map plus relevant focused references are required.
8. Run Prettier on all moved Markdown, the context structural check, `mise run dev:pi-test`, and representative manual navigation from `AGENTS.md` to every focused reference.

## Architecture impact

Application behavior does not change, but architecture documentation ownership does. The root document becomes an index and stable boundary map; focused files become semantic owners for volatile details. Any existing automation or links that assume all tables live in `docs/architecture.md` must be updated atomically. The root routing model remains backward-compatible for agents because the required entry path is unchanged.

## Performance and benchmarks

This reduces prompt/context input rather than application runtime. File reads remain linear in selected document size. The required one-off benchmark is the before/after byte and approximate-token inventory, with a 50% root-document reduction target and per-journey loaded-context totals. Keep structural size limits as regression checks only if they are generous enough to permit legitimate documentation growth; no runtime benchmark is needed.

## Cost profile

No API, infrastructure, or storage charge is added. Lower context size reduces paid model input. Record saved tokens per representative turn and calculate `saved_tokens / 1,000,000 × provider_input_price`; do not claim a currency saving without naming the model price used. More focused reads may add a small number of local tool calls but no external cost.

## Production changes

None. This is repository documentation and harness routing only. No deployment, environment variable, migration, or manual production action is required.

## Documentation updates

Restructure `docs/architecture.md`; create the focused `docs/architecture/*.md` references listed above; update `AGENTS.md` routing; tighten frontmatter in affected `.agents/skills/*/SKILL.md`; and add the context inventory and journey map to `docs/pi-harness.md`. Update cross-links in `docs/project-conventions.md` and skill bodies when their architecture anchors move.
