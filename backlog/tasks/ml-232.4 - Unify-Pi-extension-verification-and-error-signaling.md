## Description

<!-- SECTION:DESCRIPTION:BEGIN -->

Local and CI Pi verification currently enumerate different extension subsets; prod-errors and prod-logs have no suites, production tools return unsupported isError fields instead of throwing, and format-on-edit suppresses formatter failures. Establish one claim-matched verification entry point and make failures visible under the current Pi extension contract.
<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria

<!-- AC:BEGIN -->

- [ ] #1 mise run dev:pi-test is the single repository-owned Pi verification command used unchanged by developers and the Pi Extensions workflow
- [ ] #2 The command discovers or validates every testable extension and covers sensitive-file-guard, S3, CI, production logs, production errors, production metrics, mutation approval, and format-on-edit
- [ ] #3 All custom LLM tools throw on failed execution so Pi records isError true under the current tool contract
- [ ] #4 Formatter command failures preserve the successful file mutation but mark the tool result as failed with the formatter command, exit status, and actionable stderr
- [ ] #5 Network-facing extension tests use stubs and require no production credentials or live external requests
- [ ] #6 The Pi Extensions workflow installs from committed locks and invokes only mise run dev:pi-test rather than duplicating an extension list
- [ ] #7 A deliberate failing-tool and failing-formatter fixture proves CI fails and the model-visible result carries the diagnostic

<!-- AC:END -->

## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->

## Objective and chosen route

Make `mise run dev:pi-test` the only semantic owner for Pi verification and have CI call it directly. Use manifest-driven test discovery plus explicit harness-wide checks so a new extension cannot be silently omitted. Align custom-tool and formatter failures with the current Pi contract.

The rejected alternatives are keeping a hand-maintained CI list, because it has already drifted, and returning `{isError: true}` from tool `execute`, because Pi documents that only a thrown exception marks execute failure.

## Implementation steps

1. Define the extension verification contract in `scripts/dev/pi-test`: locked install, formatting or type checks where configured, each extension test script, harness-wide namespace/version checks, and the clean extension-load smoke. Make discovery fail when an extension with executable behavior has neither tests nor an explicit documented exclusion.
2. Standardize extension manifests so every suite exposes `npm test`. Add shared Node test helpers for mock `ExtensionAPI`, registered-tool lookup, environment isolation, mocked `pi.exec`, mocked UI, and fetch stubbing without starting Pi or contacting external services.
3. Add prod-errors tests for read tools, all four approved mutation tools, malformed responses, credential failures, HTTP failures, output truncation, and no-network denial paths. Add prod-logs tests for credential failures, HTTP/JSON variants, filtering, tail limits, truncation, and error signaling. Retain and extend prod-metrics tests to the same contract.
4. Audit every registered custom LLM tool. Replace error-shaped return objects with thrown `Error` instances containing bounded actionable messages. Keep cancellation as a normal non-error result only where Pi initiated cancellation. Add tests that invoke registered tools and assert promise rejection; the extension-load smoke must prove Pi converts the rejection to `isError: true`.
5. Refactor format-on-edit to inspect the `pi.exec` result rather than swallowing it. On formatter failure, return a `tool_result` middleware patch that preserves the original successful edit/write content, appends “file changed but formatting failed”, sets `isError: true`, and includes command, exit code, and bounded stderr without sensitive environment data.
6. Test successful formatting, unsupported file types, edit/write failures that skip formatting, formatter non-zero exits, thrown formatter-process errors, and cancellation. Include a deliberate failing formatter fixture to prove the model-visible result is actionable.
7. Replace `.github/workflows/pi.yml` extension-specific install/test steps with setup plus exactly `mise run dev:pi-test`. Expand path filters to include the verification script, Pi/MCP manifests and locks, settings, and workflow itself. Keep GitHub Actions pinned.
8. From a state without extension `node_modules`, run `mise run dev:pi-test` twice to prove clean install and repeatability. Run the Pi workflow locally where possible and in CI; inject no production secrets and assert test fetch stubs received every expected request.

## Architecture impact

This changes repository tooling and extension behavior only. The test runner becomes the verification owner; package manifests describe per-extension commands; CI delegates instead of duplicating. Production API/controller behavior is unchanged by this subtask. Tool callers will now receive proper Pi error results and can reason about failure reliably.

## Performance and benchmarks

Runtime changes occur only on failure paths, except one formatter exit-code check already available in the process result. Test discovery is O(E) for E extension directories and locked installs dominate CI time. Record the clean and warm `dev:pi-test` durations and keep the workflow below its existing practical timeout; no application benchmark or ongoing microbenchmark is needed.

## Cost profile

No live production, model, or paid third-party API calls are permitted. The only cost increase is several minutes of ordinary GitHub Actions execution and npm registry downloads already required by the workflow.

## Production changes

None. CI and local tests use fetch and credential-provider stubs. No production environment variable, endpoint call, migration, or deployment is required.

## Documentation updates

Update `docs/available-tasks.md` with the exact scope of `dev:pi-test`. Update `docs/pi-harness.md` with the verification contract, extension exclusion policy, failure semantics, and clean-run command. Correct `docs/project-conventions.md` only if it describes Pi extension formatting or CI checks that this work changes.
