## Description

<!-- SECTION:DESCRIPTION:BEGIN -->

Production observability currently uses one broad API bearer token, keeps credentials in the Pi process environment, and lets prod-errors mutation tools and TUI shortcuts change ErrorTracker state without a mechanical approval boundary. Introduce explicit read and mutation authority so production inspection remains convenient while every state change is interactive, auditable, least-privilege, and denied by default outside the TUI.
<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria

<!-- AC:BEGIN -->

- [ ] #1 Every prod-errors mutation tool and TUI mutation action requires an explicit TUI approval bound to the exact action and error ID before any network request is sent
- [ ] #2 Mutation attempts without approval, with mismatched or reused approval, or from print, JSON, or RPC modes are rejected without contacting production
- [ ] #3 Approval decisions are recorded in the Pi session as auditable metadata without recording credentials
- [ ] #4 Production observability reads and ErrorTracker mutations use separate server-side bearer-token scopes, and tests prove cross-scope requests are denied
- [ ] #5 Pi production credentials are resolved only when the owning tool executes and are not stored as raw ambient values in the Pi process environment
- [ ] #6 The Coolify log credential is documented and configured with the narrowest available read-only application-log scope
- [ ] #7 Automated tests cover approval, denial, non-interactive use, credential resolution, API scope enforcement, and all four mutation actions
- [ ] #8 A dedicated production-changes section documents provisioning, rollout verification, revocation, and rollback

<!-- AC:END -->

## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->

## Objective and chosen route

Replace prose-only production authority with two mechanical layers: scoped server credentials and a Pi-side one-shot approval gate. Keep the existing JSON APIs and extension UX; do not introduce a new service or database. `API_TOKEN` remains the credential for collection/asset clients, a new observability-read token protects metrics and ErrorTracker GET routes, and a separate ErrorTracker-mutation token protects the four POST routes.

Production secrets will be obtained on demand through configured credential commands (for example, macOS Keychain or another local secret broker) rather than raw `PI_*_TOKEN` values inherited by the Pi process. This removes ambient values but is not an OS sandbox: another process running as the same user may still access the same secret broker, and that residual boundary must be documented.

The rejected alternatives are: keeping one token and relying on prompts, because it does not enforce least privilege; adding confirmation only to tool descriptions, because models can bypass prose; and building a separate approval web service, because that adds deployment and availability complexity without improving this single-user harness enough to justify it.

## Implementation steps

1. Add focused controller/auth tests that describe three scopes: existing application API access, observability reads, and ErrorTracker mutations. Prove missing, wrong, and cross-scope tokens return 401 before changing routing.
2. Refactor `MusicLibraryWeb.Auth` so the bearer-token plug selects an explicitly configured token key. Add separate router pipelines for collection/assets, observability GET routes, and ErrorTracker POST routes. Load required production token values in `config/runtime.exs`; use distinct deterministic values in test config.
3. Extract production-extension credential resolution behind a small injected provider. Non-secret host/app identifiers may remain environment configuration, but bearer values must come from dedicated command references and be resolved only inside the owning operation. Reject empty output, non-zero exits, multiline output, and timeouts without including secret output in diagnostics.
4. Add a shared prod-errors authority helper. It must accept only `ctx.mode === "tui"`, display the exact environment, action, and error ID, create a short-lived one-shot grant after explicit confirmation, consume it before credential resolution or `fetch`, and reject mismatched, expired, or reused grants.
5. Persist approved and denied decisions with `pi.appendEntry` as non-LLM session audit entries containing grant ID, action, target ID, decision, mode, and timestamp. Never persist bearer values or credential-command output.
6. Route all four mutation tools and both mutation paths in `/prod-errors` through the same helper. Refactor `ErrorBrowser` to receive an async mutation callback so keyboard shortcuts cannot call `postApi` directly. Keep read-only listing/detail behavior independent of mutation authority.
7. Add Node tests with mocked Pi APIs, credential commands, and `fetch`. Cover approval/denial, non-TUI modes, one-shot binding, audit entries, resolution order (no credential lookup or network before approval), all four actions, and TUI shortcuts. Add prod-logs tests proving it asks only for the read-only Coolify credential.
8. Run focused verification, then the repository command once ML-232.4 is available: `mix test test/music_library_web/controllers/error_controller_test.exs`, the Auth tests added in step 1, `npm test --prefix .pi/extensions/prod-errors`, `npm test --prefix .pi/extensions/prod-logs`, and `mise run dev:pi-test`.

## Architecture impact

This changes API authentication pipelines and runtime configuration, but not controller response formats, contexts, schemas, databases, PubSub, or supervision. Existing collection/Presto clients continue using `API_TOKEN`. Pi metrics/ErrorTracker reads move to a read token; only mutation calls obtain the mutation token. The extension approval helper becomes the single semantic owner for interactive production mutation authority.

## Performance and benchmarks

Bearer checks remain constant-time comparisons. Each production call adds one local credential-command execution; do not cache mutation secrets across calls, because a long-lived cache would recreate ambient authority. This latency should be small compared with the production HTTP request. Record one local timing sample for credential lookup and flag sustained lookup above 500 ms, but no recurring benchmark is required. Tests make no live or paid API calls.

## Cost profile

No new hosted resource, database, or paid API is required. Runtime cost is one local secret-broker invocation per production request. CI uses stubs. Coolify token provisioning uses the existing installation; if the installed Coolify version cannot issue a read-only application-log token, stop and document the limitation rather than silently retaining a broad token.

## Production changes

1. With explicit user approval, generate independent high-entropy observability-read and ErrorTracker-mutation tokens.
2. Add the new server environment variables before deploying code that requires them; retain `API_TOKEN` for existing collection clients.
3. Provision or rotate a Coolify token to the narrowest available application/log-read scope.
4. Store Pi-side secrets in the selected local secret broker and configure only credential-command references plus non-secret host identifiers.
5. Deploy normally, then verify each token succeeds only for its intended route. Use nonexistent ErrorTracker IDs for mutation-scope authentication checks so verification cannot change state.
6. Revoke the old Pi use of the broad API/Coolify credentials after successful verification. Roll back by restoring the previous application image/routing and credential-command configuration; do not delete old server values until rollback is no longer needed.
