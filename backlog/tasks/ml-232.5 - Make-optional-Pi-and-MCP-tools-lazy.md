## Description

<!-- SECTION:DESCRIPTION:BEGIN -->

Tidewave and the full Chrome DevTools MCP catalog are exposed as direct tools, while production and CI tools are active in every session. Keep core coding tools immediately available, move MCP servers behind the existing lazy proxy, and add one small capability-search tool that additively activates optional repository tools only when a task needs them.
<!-- SECTION:DESCRIPTION:END -->

## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->

## Objective and chosen route

Keep only core coding tools plus two discovery surfaces active by default: the existing MCP proxy and a small repository tool-search loader. Configure Tidewave and Chrome DevTools as lazy proxy servers, and register CI/production tools normally but deactivate them at session start until the loader additively enables matches.

This follows Pi native dynamic-tool loading and the MCP adapter default instead of inventing separate wrappers for every tool. The rejected alternatives are keeping a hand-picked direct Chrome subset, which still adds browser schemas to non-UI work, and fixed workflow presets, which make capabilities less discoverable and require users to choose a mode before asking for work.

## Implementation steps

1. Add a deterministic inventory test that captures the current default active tools, tool-definition JSON byte size, prompt snippets/guidelines, and MCP direct-tool count. Save the measurement methodology and baseline, not volatile machine-specific absolute timings.
2. Change `.mcp.json` so Tidewave and Chrome DevTools use `lifecycle: "lazy"` and `directTools: false`. Keep the pinned Chrome command from ML-232.3. Verify the `mcp` proxy remains active and cached metadata supports search without starting either server.
3. Add a small project extension that registers `search_tools`. Define optional groups by stable tool names or source metadata: CI tools, production read tools, and production mutation tools. On `session_start`, remove those optional tools from the active set while preserving built-ins, the MCP proxy, unrelated extensions, and the loader.
4. Implement bounded local keyword matching over `pi.getAllTools()` names and descriptions. Return at most five ranked matches and call `pi.setActiveTools` with a purely additive union. Never remove tools during a loader execution. Omit active-only prompt metadata from the lazy tools where their descriptions are sufficient, to preserve prompt-cache benefits.
5. Keep production mutations as a distinct search result/group. Loading them grants capability visibility only; execution must still pass the ML-232.1 TUI approval and credential checks. Add a test that activation alone cannot issue a request.
6. Add unit tests for default selection, unknown tools, repeated searches, ranking, result bounds, additive activation, source/group isolation, and session reload. Add static MCP tests proving both servers are proxy-only and lazy.
7. Add representative smoke journeys: core file inspection needs no discovery; UI work discovers and invokes Chrome through `mcp`; database/runtime work discovers Tidewave through `mcp`; production health activates read tools but not mutations; CI monitoring activates CI tools. Use fake server/tool implementations in automation and document one manual real-server check for Chrome and Tidewave.
8. Repeat the inventory after the change. Require at least 30 fewer default direct/optional tools and at least a 40% reduction in serialized default tool-schema bytes, or explain with measured evidence why an adjusted threshold is sound. Run the extension tests and `mise run dev:pi-test`.

## Architecture impact

No application architecture changes. `.mcp.json` owns MCP lifecycle/exposure; the new loader owns default activation for repository extension tools; each original extension continues to own execution. Existing tool names and slash commands remain available. This adds one discovery turn for the first optional capability in a session.

## Performance and benchmarks

Tool search is local O(T × Q), where T is registered tool count and Q is query terms, with a small bounded result. MCP servers no longer start for unrelated work. On native deferred-tool models, additive activation preserves the prompt prefix; other models receive the expanded active list on the next turn. A first optional journey costs one extra local tool call/model round trip, while every prior request avoids roughly 35 or more MCP schemas plus CI/production schemas. The before/after schema inventory is a required one-off benchmark; keep a regression assertion on default active names, not a timing benchmark.

## Cost profile

Discovery itself makes no external request. It may add one model round trip on the first UI, production, or CI journey. Pi MCP documentation estimates roughly 150 to 300 prompt tokens per direct tool, so removing at least 30 default definitions saves about 4,500 to 9,000 input tokens per request before the additional CI/production savings. Record the actual schema-byte/token estimate and apply the active model price formula in `docs/pi-harness.md`; the recurring savings should exceed the one-time loader turn in multi-turn sessions.

## Production changes

None. Production tools remain local clients and tests use stubs. No server variables, provisioning, deployment, or live production verification is required for this subtask.
