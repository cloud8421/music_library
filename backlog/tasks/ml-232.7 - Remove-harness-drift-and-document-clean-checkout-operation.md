## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->

## Objective and chosen route

Finish the initiative by removing duplicate volatile claims, correcting every known mismatch, and publishing one operational Pi harness guide backed by a clean-checkout CI smoke. Prefer deleting duplicated facts and linking to an owner over adding parsers or generators; add a mechanical consistency check only where duplication is necessary for an executable skill.

The rejected alternative is a broad documentation generator for Elixir config, shell, TypeScript, and Markdown. That would be more code than the drift it prevents and would create another semantic owner.

## Implementation steps

1. Build a fact-owner matrix for schedules, Pi/MCP versions, active tool names, verification coverage, production authority, and pre-commit behavior. For each fact choose one executable owner and list every duplicate reference that must be removed, linked, or checked.
2. Make `config/prod.exs` the RepoVacuum schedule owner. Update the current 03:00 architecture entry to 03:03, then remove exact cron duplication from general architecture where the focused background reference can link to config. If the Oban skill retains a convenience table, add a small consistency check for its values.
3. Update QueryReporter instructions for the post-ML-232.5 tool surface: discover Tidewave through `mcp`, then call the documented project-eval and SQL tools through the proxy. Include the direct-tool fallback only if it is actually supported by the configured epoch. Verify the start, trigger, stop, and read flow against a running dev server.
4. Replace the documentation skill requirement for unavailable subagents with explicit parallel `read`/`bash` tool calls through the available parallel wrapper, with a sequential fallback. Preserve its user-approval requirement and generated usage-rules guard.
5. Treat `scripts/dev/precommit` as the pre-commit owner. Correct `docs/project-conventions.md` claims about AGENTS.md triggers, Sobelow flags, dependency audit behavior, Docker triggers, and any other command mismatch. Prefer a concise generated `--describe` output or a consistency fixture only if the detailed table remains necessary.
6. Create or consolidate `docs/pi-harness.md`. Cover the exact worker epoch and update process, project trust, sensitive-file boundary and exclusions, production credential scopes and approval audit, default/lazy tools, verification command, smoke journeys, context measurements, failure semantics, and troubleshooting. Include links rather than repeating volatile version/schedule/tool lists owned elsewhere.
7. Add a clean-checkout harness smoke script with two phases: locked registry installation, then `PI_OFFLINE=1` runtime verification in a temporary Pi agent/session directory. It must report versions, validate no forbidden floating references/legacy imports, load every extension, run `mise run dev:pi-test`, inspect default tool activation, and avoid provider authentication, production credentials, browsers, or live MCP servers.
8. Make `.github/workflows/pi.yml` run the smoke from a clean checkout after the unified suite. Use explicit timeouts and no secrets. Add an evidence matrix mapping ML-232 AC #1-#10 and every child criterion to an automated command, a manual production step from ML-232.1, or a documented exclusion.
9. Run `mise run dev:pi-test`, the clean smoke from removed dependency directories or an equivalent temporary clone, Markdown formatting, shellcheck for changed scripts, and targeted link/consistency checks. Review CI evidence before marking any parent criterion complete.

## Architecture impact

No application behavior changes are intended. Documentation ownership changes: executable config/scripts/manifests own volatile facts; `docs/pi-harness.md` owns the supported operating model; focused architecture and skills link to those owners. CI becomes proof of clean installation and extension loading, not merely unit tests.

## Performance and benchmarks

The smoke adds setup time proportional to locked npm packages and extension suites but no production runtime overhead. Record clean versus cache-warm CI duration and set a timeout that catches hangs without encoding workstation speed. No application benchmark is required. Retain the tool/context measurements from ML-232.5 and ML-232.6 as one-off qualification evidence rather than rebenchmarking every CI run.

## Cost profile

CI consumes normal GitHub Actions minutes and registry bandwidth. Runtime smoke is offline and makes zero model, production, Coolify, or paid API calls. No new hosted service or storage is provisioned.

## Production changes

No new manual production change belongs to this final subtask. `docs/pi-harness.md` must link to the dedicated rollout and rollback procedure delivered by ML-232.1. Do not execute those steps, fetch production data, or rotate credentials while completing this documentation/CI work.

## Documentation updates

Create/finalize `docs/pi-harness.md`; correct `docs/project-conventions.md`, `docs/architecture.md` or its focused background/tooling references, `.agents/skills/query-reporter/SKILL.md`, `.agents/skills/update-documentation/SKILL.md`, and `.agents/skills/oban-worker/SKILL.md` as dictated by the owner matrix. Update `docs/available-tasks.md` for the smoke/verification tasks and `docs/production-infrastructure.md` only to link the final authority/credential sections without duplicating them.
