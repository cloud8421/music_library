## Description

<!-- SECTION:DESCRIPTION:BEGIN -->

A clean checkout cannot currently reproduce the Pi runtime: Pi itself is not managed by mise, the project package spec is floating, its lockfile is ignored, Chrome DevTools MCP runs at latest, and extension imports mix legacy and current package namespaces. Define and qualify one exact worker epoch that can be installed consistently on supported macOS and Linux environments.
<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria

<!-- AC:BEGIN -->

- [ ] #1 mise.toml and mise.lock pin an exact Earendil Works Pi coding-agent release for every supported platform
- [ ] #2 The Pi MCP adapter, Chrome DevTools MCP server, and repository-owned extension dependency trees are fixed by committed exact package specs and lockfiles
- [ ] #3 No supported harness configuration uses latest, an unqualified package source, or an ignored lockfile as its reproducibility mechanism
- [ ] #4 All repository Pi extensions import only the supported @earendil-works package namespace
- [ ] #5 A clean-checkout qualification installs the epoch, loads every project extension, reports the selected versions, and fails on version or namespace drift
- [ ] #6 The qualification runs on the supported CI platform and documents any platform-specific limitations for local macOS use

<!-- AC:END -->

## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->

## Objective and chosen route

Make the repository, not workstation history, define the Pi worker epoch. Start from the currently qualified runtime (`@earendil-works/pi-coding-agent` 0.73.1 and `pi-mcp-adapter` 2.11.0), select one exact Chrome DevTools MCP release during qualification, and commit every direct and transitive npm resolution needed by the harness. Use only the Earendil Works package namespace.

The rejected alternatives are Dockerizing all Pi use, which would complicate browser and Tidewave access, and pinning only top-level versions while allowing transitive re-resolution, which does not reproduce the tested epoch.

## Implementation steps

1. Inventory the current executable and package graph: Pi, Node, MCP adapter, Chrome DevTools MCP, TypeBox, extension-local dependencies, and every legacy import. Record the baseline versions before changing manifests.
2. Add exact `npm:@earendil-works/pi-coding-agent` 0.73.1 tooling to `mise.toml` and regenerate `mise.lock` for the configured macOS and Linux platforms. Confirm the installed `pi --version` comes from mise rather than an unrelated global binary.
3. Replace the floating adapter source in `.pi/settings.json` with `npm:pi-mcp-adapter@2.11.0`. Make `.pi/npm/package.json` and `package-lock.json` repository-owned, change its ignore rules to keep only installed `node_modules` out of Git, and use exact direct versions without caret or tilde ranges.
4. Resolve and qualify one Chrome DevTools MCP version, add it to the committed harness lock, and invoke the locked local binary from `.mcp.json` instead of `npx -y ...@latest`. Keep the MCP lifecycle configuration independent of this pin; ML-232.5 owns direct-versus-proxy exposure.
5. Give every extension with third-party dependencies an exact package manifest and committed lock. Add minimal manifests and locks for currently untested extensions as needed by ML-232.4. Do not commit installed dependency directories.
6. Replace every `@mariozechner/pi-*` import with the corresponding `@earendil-works/pi-*` import and update test-time dynamic imports. Add a repository check that fails on the legacy namespace, `@latest`, unversioned Pi package sources, or unlocked harness manifests.
7. Add a clean-checkout epoch qualification script. In a temporary or clean package state it must run locked installs, report all selected versions, import and load each project extension without provider credentials, and use `PI_OFFLINE=1` for the runtime phase so Pi performs no model, telemetry, update, or package-update network operations.
8. Run `mise install`, locked npm installs, every extension test available at this point, the namespace/version check, and the epoch smoke on macOS. Add the same qualification to Ubuntu CI; treat platform-specific browser-launch limitations as documented exclusions, not silent skips.

## Architecture impact

This changes developer-tool provisioning only. It does not alter the Phoenix release, application dependencies, databases, routes, or production container. `.pi/npm/package-lock.json` becomes the authoritative transitive npm graph for shared harness packages; extension-local lockfiles own their isolated graphs; `mise.toml` and `mise.lock` own executable versions.

## Performance and benchmarks

Pinned versions should not materially change runtime behavior. Replacing `npx` with the locked local Chrome binary removes package-resolution startup work and should improve first-call latency. The clean qualification adds CI time but no application runtime cost. Capture one before and after cold launch timing for the Chrome MCP command and Pi extension load; no recurring performance benchmark is required beyond a CI timeout that detects hangs.

## Cost profile

There are no paid API calls or new hosted resources. CI downloads npm and mise artifacts and consumes ordinary workflow minutes. Lockfile growth is repository storage only.

## Production changes

None. Pi and MCP tooling run in the development harness and are not included in the deployed Phoenix image. No production environment variables, migrations, or provisioning are required.

## Documentation updates

Add the exact worker epoch, supported platforms, installation command, update procedure, and qualification command to `docs/pi-harness.md`. Update `docs/production-infrastructure.md` only if its Tool Versions section currently implies Pi is production runtime tooling; otherwise keep the new details in the harness guide. Update `docs/available-tasks.md` if a new mise qualification task is added.
