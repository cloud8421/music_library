## Description

<!-- SECTION:DESCRIPTION:BEGIN -->

The sensitive-file guard currently fails open when its configuration cannot be loaded, overstates its coverage as all-tool protection, and tests only shell command extraction. Make the guard fail closed across its declared Pi tool and user-shell boundary, test the extension behavior end to end, and document the limits imposed by Pi extensions running with full local-user permissions.
<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria

<!-- AC:BEGIN -->

- [ ] #1 Missing, malformed, or schema-invalid guard configuration leaves a fail-closed handler active and blocks guarded operations with an actionable diagnostic
- [ ] #2 The guard enforces canonical protected paths across each supported built-in path tool, the bash tool, and user-entered shell commands
- [ ] #3 Tests exercise the registered extension handlers, including path tools, shell tools, user shell, relative and absolute paths, symlink aliases, blocked command parsing, and invalid configuration
- [ ] #4 The injected prompt and project documentation state the exact enforced boundary and explicitly exclude arbitrary extension code and external processes from the security claim
- [ ] #5 Blocked results contain no protected content and do not permit fallback execution through another guarded entry point
- [ ] #6 The complete sensitive-file-guard test suite runs through the repository-owned Pi verification command

## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->

## Objective and chosen route

Turn sensitive-file-guard into a fail-closed policy layer for the Pi entry points it can actually intercept. Keep it as an extension rather than claiming sandbox isolation. The supported boundary is built-in path tools, the model-facing bash tool, and user `!`/`!!` shell execution; arbitrary extension code, child processes started outside those entry points, and same-user OS access remain outside the boundary.

The rejected alternatives are an OS sandbox, which is valuable but materially larger than correcting this guard, and overriding every built-in tool, which duplicates Pi implementations and is more fragile than the documented `tool_call` and `user_bash` interception APIs.

## Implementation steps

1. Split configuration parsing, path matching, shell-policy evaluation, and Pi handler registration into testable modules. Validate the complete JSON shape and reject unknown or non-string policy entries that could weaken matching.
2. Register handlers before applying loaded policy state. Start in a deny state; only transition to configured enforcement after successful parsing and matcher construction. Missing or invalid configuration must keep file and shell operations blocked and explain the configuration error without exposing file content.
3. Define canonical path matching once. Compare both the normalized requested path and a realpath-canonical form; for nonexistent write targets, canonicalize the nearest existing parent before appending the remaining path. Anchor glob-derived regexes so unrelated path substrings do not match accidentally.
4. Apply the matcher to every supported built-in path tool (`read`, `write`, `edit`, `grep`, `find`, and `ls`) and to path fragments in `bash`. Preserve structural blocked-command parsing, but make parse failure conservative and deterministic.
5. Intercept `user_bash` as a separate entry point and return a failed shell result when policy blocks execution. Share the same evaluator with model-facing bash so the two paths cannot drift.
6. Replace the injected all-tool claim with exact boundary language and the project-trust/same-user caveat from Pi security documentation. Keep blocked diagnostics instructive, but do not rely on the model obeying “do not retry” as enforcement.
7. Add extension-level tests that instantiate the factory with a mock `ExtensionAPI`, capture registered handlers, and execute representative events. Cover each path tool, model bash, user bash, comments/quoted command names, nested command substitutions, relative/absolute paths, existing and nonexistent targets, symlink aliases, case handling, invalid config, and allowed controls.
8. Verify with `npm test --prefix .pi/extensions/sensitive-file-guard`; after ML-232.4, also run `mise run dev:pi-test`. Include a negative fixture that would read through a symlink if canonicalization were removed.

## Architecture impact

Only the project Pi extension boundary changes. Application modules, databases, routes, and production infrastructure are unaffected. The policy matcher becomes the authoritative implementation; prompt text and `docs/pi-harness.md` describe that implementation rather than defining additional policy.

## Performance and benchmarks

Path calls perform O(P × L) matching for P protected patterns and path length L, plus at most one filesystem canonicalization. Bash calls retain one AST parse proportional to command length. The current policy has one path and two command entries, so overhead is negligible relative to file I/O or process startup. No recurring benchmark is required; add one optional one-off test over 10,000 short path/command evaluations and investigate only if it exceeds one second on a development machine.

## Cost profile

No paid service, API request, persistent storage, or additional CI service is introduced. Test execution is local Node CPU only.

## Production changes

None. This guard runs in the local Pi harness and does not require deployment, environment variables, provisioning, or data migration.

## Documentation updates

Create the sensitive-file boundary section in `docs/pi-harness.md` (or leave a clearly identified section for ML-232.7 to consolidate), citing Pi's no-built-in-sandbox model. Update the guard's injected system-prompt text and comments so they make the same claim. Do not describe this extension as protection from arbitrary trusted extensions or same-user processes.
